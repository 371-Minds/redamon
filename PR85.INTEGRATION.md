# PR #85 Post-Merge Integration Plan: Chat Skills (`/skill`)

## Context

PR #85 introduces "Infosec Specialist Skills" -- 36 markdown files injected into the agent system prompt via project settings toggles. After analysis, we decided to **migrate this feature to a different delivery model**: on-demand injection via chat command (`/skill`) instead of always-on project-setting toggles.

The new feature is called **Chat Skills** -- tactical reference docs the user injects into the agent's prompt on the fly during a pentest session, using `/skill <name>` in the chat. This is fundamentally different from **Agent Skills** (the existing system), which are phase-aware attack workflows that drive classification and tool routing.

---

## Naming Convention

| System | Name | Command | Purpose |
|--------|------|---------|---------|
| Existing (unchanged) | **Agent Skills** | N/A (project settings toggle) | Phase-aware attack workflows. Drive classification, tool selection, and step-by-step execution across informational/exploitation/post-exploitation phases. |
| New (this plan) | **Chat Skills** | `/skill <name>` in chat | On-demand reference docs. Tactical knowledge (tool playbooks, vulnerability guides, framework notes) injected into the current agent reasoning chain when the user needs it. |

## What to ADD (New Integration)

### Phase 1: Backend -- Chat Skill delivery via guidance_messages

#### 1.1 New WebSocket message type: `SKILL_INJECT`

**File**: `agentic/websocket_api.py`

Add a new handler `handle_skill_inject` that:
1. Receives `{ "skill_id": "vulnerabilities/ssrf" }` from the frontend
2. Calls `load_skill_content(skill_id)` to get the markdown content
3. Formats it as a guidance message:
   ```
   [CHAT SKILL: SSRF Testing Guide]
   <full markdown content>
   ```
4. Pushes it onto `connection.guidance_queue` (the existing guidance pipeline)
5. Sends `SKILL_INJECT_ACK` back to the frontend with skill name and queue position

**Why guidance_messages**: The existing pipeline in `think_node.py` (lines 393-404) already:
- Drains the guidance queue each iteration
- Appends content as `## USER GUIDANCE (IMPORTANT)` section
- Is scoped to the current reasoning chain (not persistent)
- Has the right priority in the prompt stack

No changes needed to `think_node.py` -- the existing guidance injection handles it.

#### 1.2 Also support `/skill` when agent is NOT running

When the agent is idle (no active session), `/skill <name>` should still work. In this case:
- The frontend intercepts the `/skill` command before sending as a query
- Pre-loads the skill content into the next query's context
- When the user sends their next message, the skill content is included as a preamble

**Implementation**: Add a `pendingSkillContent` state in the chat. When the user sends a query and `pendingSkillContent` is set, prepend it to the query message, then clear the state.

#### 1.3 New WebSocket message type in types

**File**: `webapp/src/lib/websocket-types.ts`

```python
@app.get("/community-skills", tags=["System"])
async def list_community_skills():
    """Return catalog of community Agent Skills from agentic/community-skills/."""
    from pathlib import Path
    skills_dir = Path(__file__).parent / "community-skills"
    skills = []
    if skills_dir.exists():
        for md_file in sorted(skills_dir.glob("*.md")):
            if md_file.name == "README.md":
                continue
            content = md_file.read_text(encoding="utf-8")
            name = md_file.stem.replace("_", " ").title()
            # Extract first non-empty, non-heading line as description
            desc = ""
            for line in content.splitlines():
                stripped = line.strip()
                if stripped and not stripped.startswith("#"):
                    desc = stripped[:200]
                    break
            skills.append({
                "id": md_file.stem,
                "name": name,
                "description": desc,
                "file": str(md_file),
            })
    return {"skills": skills, "total": len(skills)}
```

#### 5.1 Add name-based lookup to skill_loader.py

**File**: `agentic/skill_loader.py`

Add a function to resolve a skill by partial name match (for the chat command):
```python
def find_skill_by_query(query: str) -> Optional[dict]:
    """Find a skill by partial name/id match. Returns the best match or None."""
    skills = list_skills()
    query_lower = query.lower().strip()
    # Exact id match first
    for s in skills:
        if s['id'] == query_lower:
            return s
    # Exact name match
    for s in skills:
        if s['name'].lower() == query_lower:
            return s
    # Partial match
    matches = [s for s in skills if query_lower in s['id'] or query_lower in s['name'].lower()]
    return matches[0] if len(matches) == 1 else None
```

async def search_skills(q: str = Query(...)):

## Agent Skills vs Chat Skills

RedAmon has two distinct skill systems. Understanding the difference is important
for getting the best results from the agent.

| | Agent Skills | Chat Skills |
|---|---|---|
| **What they are** | Attack workflows with phase-structured steps | Reference docs with tactical knowledge |
| **How activated** | Automatically -- the Intent Router classifies your request and selects the best matching skill | Manually -- you type `/skill <name>` in the chat when you need it |
| **Phase aware** | Yes -- different prompts for informational, exploitation, and post-exploitation phases | No -- the full document is injected as-is regardless of phase |
| **Affects classification** | Yes -- skills compete as classification candidates; the agent picks the best match | No -- classification is unaware of Chat Skills |
| **Tool routing** | Yes -- each skill enables/disables specific tools (sqlmap, hydra, metasploit, etc.) | No -- no impact on tool availability |
| **RoE integration** | Yes -- skills are excluded when Rules of Engagement forbid them | No -- always available regardless of RoE |
| **Persistence** | Active for the entire session (all iterations of the think loop) | Active for the current reasoning chain only |
| **Storage** | Per-user in database (uploaded .md files with workflow steps) | Per-user in database (uploaded .md reference docs) |
| **Where configured** | Global Settings > Agent Skills tab, then toggled per project | Global Settings > Chat Skills tab |
| **Token cost** | Efficient -- only the classified skill is injected | Efficient -- only injected when you ask for it |
| **Best for** | Complete attack methodologies: "Here is exactly how to exploit XSS step by step" | Quick reference: "Here is how ffuf flags work" or "Here are SSRF test vectors" |

**Rule of thumb**: If your skill defines a step-by-step attack procedure with phases,
upload it as an **Agent Skill**. If it is a reference doc, cheat sheet, or conceptual guide,
upload it as a **Chat Skill**.

## Using Chat Skills

### The /skill Command

During a pentest session, type `/skill` followed by the skill name or keyword:

```
/skill ssrf
/skill ffuf
/skill graphql

> [Chat Skill Loaded: SSRF Testing Guide]

> The agent now has SSRF testing expertise for this reasoning chain.

You can inject multiple skills in sequence:

/skill ssrf

```

The agent now has both SSRF and Next.js knowledge for its current task.

1. Navigate to **Global Settings > Chat Skills**
2. Click **Upload Skill (.md)**
3. Select your Markdown file
4. Enter a **name**, optional **description**, and **category**
5. Click **Upload**

The skill is immediately available via `/skill <name>` in all your sessions.

### Writing a Chat Skill File

Chat Skill files are simpler than Agent Skill files. They do not need phase-structured
workflows or classification keywords. Just write clear, actionable reference content:

```markdown
---
name: ffuf
description: ffuf fuzzing syntax with matcher/filter strategy and non-interactive defaults.
---

# ffuf CLI Playbook

Canonical syntax:
ffuf -w <wordlist> -u <url_with_FUZZ> [flags]

High-signal flags:
- -u <url> target URL containing FUZZ
- -w <wordlist> wordlist input
- -mc <codes> match status codes
- -ac auto-calibration
- -noninteractive disable interactive console mode

Agent-safe baseline:
ffuf -w wordlist.txt -u https://target.tld/FUZZ -mc 200,301,302,403 -ac -t 20 -noninteractive

Common patterns:
- Basic path fuzzing: ffuf -w wordlist.txt -u https://target.tld/FUZZ -mc 200,301,403 -ac
- Vhost fuzzing: ffuf -w vhosts.txt -u https://target.tld -H 'Host: FUZZ.target.tld' -fs 0
```

**Key differences from Agent Skill files:**
- No phase structure needed (no "Phase 1: Reconnaissance", "Phase 2: Exploitation")
- No classification keywords needed (no "When to Classify Here" section)
- Optional YAML frontmatter with `name` and `description`
- Focus on reference material: flags, patterns, examples, test vectors

## Jhaddix Wordlist: Bug Fix + Wordlist Selector UI

### The bug

The PR's wordlist mount logic in `recon/domain_recon.py` is broken. It derives a host path from `HOST_RECON_OUTPUT_PATH`, then checks `os.path.isfile(wordlist_host_path)` -- but that call runs **inside the recon container** using a **host filesystem path**. The path doesn't exist inside the container, so the check always returns `False` and the wordlist is never mounted. Amass silently falls back to its built-in ~8K list.

### The fix

The wordlist file is baked into the `redamon-recon` image at `/app/recon/wordlists/jhaddix-all.txt` (Dockerfile). The `recon_orchestrator` already auto-detects host paths by inspecting its own Docker mounts (`_detect_host_mounts()` in `recon_orchestrator/api.py:40`). The recon container is volume-mounted as `./recon:/app/recon:ro` (docker-compose.yml:269), so the host path for the wordlist is `<project_root>/recon/wordlists/jhaddix-all.txt`.

**Fix in `recon/domain_recon.py`**: Replace the broken host-path derivation with:

```python
# The wordlist is inside this container at /app/recon/wordlists/
# For the Amass sibling container, we need the HOST path.
# HOST_RECON_OUTPUT_PATH points to <host_project>/recon/output
# so the wordlist host path is <host_project>/recon/wordlists/jhaddix-all.txt
container_wordlist = '/app/recon/wordlists/jhaddix-all.txt'
if host_recon_output:
    # HOST_RECON_OUTPUT_PATH = /home/user/redamon/recon/output
    # -> parent = /home/user/redamon/recon
    # -> wordlist = /home/user/redamon/recon/wordlists/jhaddix-all.txt
    host_recon_dir = os.path.dirname(host_recon_output)
    wordlist_host_path = os.path.join(host_recon_dir, 'wordlists', 'jhaddix-all.txt')
else:
    wordlist_host_path = ''

# Check that the file exists INSIDE this container (not on the host)
wordlist_available = os.path.isfile(container_wordlist)
```

Then use `wordlist_available` (not `os.path.isfile(wordlist_host_path)`) for the mount decision, and use `wordlist_host_path` for the `-v` bind mount path.

#### Backend logic

**File**: `recon/domain_recon.py`

Update `run_amass()` to read the wordlist selection:

```python
brute_wordlists = settings.get('AMASS_BRUTE_WORDLISTS', ['default'])

if brute:
    command.append('-brute')

    # 'default' = Amass built-in wordlist (no -w flag needed, always active)
    # 'jhaddix-all' = jhaddix all.txt, bind-mounted from host into Amass container
    if 'jhaddix-all' in brute_wordlists:
        container_wordlist = '/app/recon/wordlists/jhaddix-all.txt'
        if os.path.isfile(container_wordlist) and wordlist_host_path:
            command_insert_idx = command.index(docker_image)
            command.insert(command_insert_idx, f'{wordlist_host_path}:/wordlist/jhaddix-all.txt:ro')
            command.insert(command_insert_idx, '-v')
            command += ['-w', '/wordlist/jhaddix-all.txt']
            print(f"[*][Amass] Using jhaddix all.txt wordlist (~2.18M entries) for brute force")
        else:
            print(f"[!][Amass] jhaddix wordlist selected but not found at {container_wordlist} -- using Amass default")
    else:
        print(f"[*][Amass] Using Amass built-in wordlist (~8K entries) for brute force")
```
